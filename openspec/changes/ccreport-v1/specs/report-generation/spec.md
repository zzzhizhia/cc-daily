## ADDED Requirements

### Requirement: Generate report header from metadata
The system SHALL generate the tweetable summary line and statistics overview table deterministically from extracted metadata (not from LLM). The header format SHALL be:
- Line 1: `M/DD: N个项目 · M个session · 主要推进 <top projects>`
- Table: 项目数, Session 数, 消息数, 活跃时段, Focus

#### Scenario: Header generation
- **WHEN** metadata shows 5 projects, 12 sessions, 47 messages, focus on ccreport (70%), active 14:00-18:00
- **THEN** header includes tweetable line and correctly populated statistics table

### Requirement: Construct prompt with metadata context
The system SHALL inject metadata context into the prompt sent to claude -p. The prompt SHALL include a context line ("今天有 N 个项目、M 个 session、K 条消息") and instruct claude -p to generate only project detail bullets (not tweetable or stats).

#### Scenario: Single mode prompt
- **WHEN** total estimated tokens < TOKEN_BUDGET
- **THEN** a single prompt is constructed containing all project texts with metadata context

#### Scenario: Split mode prompt
- **WHEN** generating for a single project in split mode
- **THEN** the prompt contains only that project's text and instructs output starting with `## <project path>`

### Requirement: Budget-aware three-level splitting
The system SHALL estimate tokens as `total_chars / 2`. TOKEN_BUDGET SHALL be a configurable constant (default 150,000). The system SHALL implement three levels:
- L1: total < TOKEN_BUDGET → single claude -p call
- L2: total >= TOKEN_BUDGET → split by project, parallel calls (concurrency max 3)
- L3: single project >= TOKEN_BUDGET → split that project by session groups

#### Scenario: Under budget
- **WHEN** total estimated tokens is 80,000
- **THEN** a single claude -p call is made with all project texts

#### Scenario: Over budget, split by project
- **WHEN** total estimated tokens is 200,000 across 4 projects
- **THEN** each project gets its own claude -p call, with at most 3 running concurrently

#### Scenario: Single project over budget
- **WHEN** one project alone has estimated tokens of 180,000
- **THEN** that project's sessions are grouped into chunks under TOKEN_BUDGET and called separately

#### Scenario: Single call failure triggers fallback
- **WHEN** a single-mode claude -p call fails (non-zero exit)
- **THEN** the system automatically retries by splitting into per-project calls

### Requirement: Claude -p execution with error handling
The system SHALL invoke `claude -p` via `spawn` with stdin stream (not execSync). Each subprocess SHALL have a 120-second independent timeout. The system SHALL detect rate-limit responses (429 or keyword match in stderr) and handle them separately from other failures.

#### Scenario: Successful execution
- **WHEN** claude -p returns exit code 0
- **THEN** stdout is captured as the generated markdown

#### Scenario: Timeout
- **WHEN** a claude -p subprocess exceeds 120 seconds
- **THEN** the process is killed and treated as a failure

#### Scenario: Rate limit detection
- **WHEN** parallel calls receive rate-limit errors
- **THEN** concurrency reduces to 1 and exponential backoff is applied before retrying

#### Scenario: Claude not installed
- **WHEN** the `claude` command is not found
- **THEN** the system exits with error: "Claude Code not found"

### Requirement: Merge split results
The system SHALL merge results from parallel calls by sorting project markdown sections by project name and concatenating them. The header (tweetable + stats) generated by formatter SHALL be prepended.

#### Scenario: Merge three project results
- **WHEN** three parallel calls return markdown for projects A, B, C
- **THEN** final output is header + project A section + project B section + project C section (sorted)
